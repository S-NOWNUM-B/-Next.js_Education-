# Базовая структура Next.js 15: App Router и архитектура

Next.js 15 - это full-stack фреймворк на базе React 19, где основной способ организации приложения строится вокруг App Router. Маршруты, макеты, состояния загрузки и обработка ошибок задаются через файловую структуру, что делает архитектуру проекта предсказуемой и удобной для масштабирования.

## Что такое App Router

App Router - это система маршрутизации, в которой путь страницы определяется структурой папок внутри директории app.

- Каждый сегмент URL соответствует папке.
- Специальные файлы внутри сегмента управляют его поведением.
- По умолчанию компоненты считаются серверными (Server Components).

## Файловый роутинг и ключевые файлы

Внутри app используются зарезервированные файлы:

- page.tsx: UI конкретного публичного маршрута.
- layout.tsx: общий каркас для сегмента и его дочерних страниц; не перерисовывается полностью при навигации.
- loading.tsx: fallback-интерфейс загрузки на основе Suspense.
- error.tsx: клиентская граница ошибок для сегмента.

### Минимальный пример структуры

```txt
src/
    app/
        layout.tsx
        page.tsx
        dashboard/
            layout.tsx
            page.tsx
            loading.tsx
            error.tsx
```

## Server Components и Client Components

В App Router компоненты по умолчанию серверные.

- Server Components:
    - выполняются на сервере;
    - не отправляют лишний JavaScript в браузер;
    - могут обращаться к БД и приватным данным на сервере.
- Client Components:
    - нужны для интерактивности (события, useState, useEffect);
    - объявляются через директиву "use client" в начале файла.

### Пример клиентского компонента

```tsx
"use client";

import { useState } from "react";

export default function Counter() {
    const [count, setCount] = useState(0);

    return (
        <button onClick={() => setCount((prev) => prev + 1)}>
            Счетчик: {count}
        </button>
    );
}
```

## Асинхронные API в Next.js 15

В версии 15 ряд API стал асинхронным. Это важное изменение, которое требует использования await.

К асинхронным относятся:

- cookies() и headers();
- params и searchParams в page/layout/generateMetadata;
- draftMode().

### Пример страницы с async params

```tsx
type PageProps = {
    params: Promise<{ slug: string }>;
};

export default async function ArticlePage({ params }: PageProps) {
    const { slug } = await params;

    return <h1>Статья: {slug}</h1>;
}
```

## Server Actions

Server Actions - это асинхронные функции с директивой "use server", которые позволяют выполнять серверную логику напрямую из UI (например, обработку формы и запись в БД).

Преимущества:

- Безопасность: Next.js генерирует неугадываемые ID действий и исключает неиспользуемые actions из клиентского бандла.
- Прогрессивное улучшение: формы могут работать даже до полной загрузки клиентского JavaScript.

### Пример Server Action

```tsx
export async function createPost(formData: FormData) {
    "use server";

    const title = formData.get("title");

    if (typeof title !== "string" || title.trim().length < 3) {
        throw new Error("Некорректный заголовок");
    }

    // Здесь обычно выполняется запись в БД.
}
```

## Изменения в кешировании (v15)

По умолчанию в Next.js 15 больше не кешируются:

- GET Route Handlers;
- клиентский роутер-кеш сегментов страниц (staleTime = 0);
- fetch-запросы без явной настройки кеша.

Это делает поведение более предсказуемым для динамических данных, но требует осознанно настраивать кеширование там, где оно действительно нужно.

## Практические правила при разработке

- Четко разделяйте серверную и клиентскую логику через "use server" и "use client".
- Валидируйте входные данные в Server Actions (например, через Zod).
- Не полагайтесь только на Middleware для авторизации; проверяйте доступ в серверном слое данных.

## Заключение

Понимание структуры App Router в Next.js 15 - фундамент для разработки производительных и безопасных приложений. Базовая практика: правильно разложить проект по файловым соглашениям, использовать Server Components по умолчанию, а Client Components подключать только там, где действительно нужна интерактивность.
